# Data Model: Compliance Reporting Maintenance

## Candidate Demographic Field Set

Represents optional candidate-provided demographic disclosure used only for aggregate D&I reporting.

**Fields**: candidate id, gender category, ethnicity category, disability category, veteran or protected status category if used by the team, consent flag, disclosure updated timestamp, withdrawn timestamp.

**Relationships**: One candidate may have zero or one demographic field set. Reports aggregate across applications and requisitions through the candidate's applications.

**Validation**: All fields are optional. Stored values must come from approved category lists or be recorded as "Not provided". Candidate profile updates must allow withdrawal or blank disclosure. HR report views must never display individual demographic values.

**State Transitions**: Not provided -> Provided; Provided -> Updated; Provided -> Withdrawn; Withdrawn -> Provided.

## Pipeline Throughput Report

Aggregate read model for HR analytics.

**Fields**: report scope, date range, requisition id, department id, stage, candidate count, entered stage timestamp, exited stage timestamp, average stage days, conversion rate, time-to-hire, bottleneck flag, bottleneck reason.

**Relationships**: Uses job requisitions, departments, applications, application status histories, interviews, feedback, offers, and onboarding state.

**Validation**: Date range must be valid and bounded. Requisition and department filters must refer to visible records. Empty scopes return an empty-state result rather than an error.

## D&I Audit Report

Aggregate read model for privacy-safe diversity reporting.

**Fields**: report scope, date range, category type, category value, application stage, outcome, raw count, displayed count, suppressed flag, suppression reason, generated by, generated at.

**Relationships**: Aggregates candidate demographic field sets through applications, job requisitions, departments, and outcomes.

**Validation**: Groups with fewer than 3 candidates must be suppressed or combined. Non-disclosing candidates must be included as "Not provided". Report generation requires HR Admin authorization.

## Compliance Run Check Batch

Represents one HR-triggered maintenance execution.

**Fields**: batch id, check type, selected scope, actor user id, status, started at, completed at, total findings, new notifications, duplicate notifications skipped, archive recommendations, blocked actions, summary message.

**Relationships**: One batch has many run-check findings. A batch may create notification escalations and archive candidates.

**Validation**: Check type must be one of missing feedback, offer expiry, simulated background check, onboarding tasks, archive closed requisitions, archive rejected candidates, or all checks. Actor must be active HR Admin. Scope filters must be valid.

**State Transitions**: Started -> Completed; Started -> Failed. Failed runs retain error summary and no partial archive approvals.

## Compliance Run Check Finding

Represents an actionable, skipped, duplicate, or informational result from a batch.

**Fields**: finding id, batch id, finding type, severity, entity type, entity id, candidate id if applicable, responsible user id, due date, detected at, recommended action, existing notification id, created notification id, archive eligibility status, reason, resolved marker.

**Relationships**: Belongs to a run-check batch. May reference users, candidates, applications, interviews, offers, onboarding records, notifications, or archive actions.

**Validation**: Entity references must match finding type. Duplicate escalation prevention must check open notification reference type and id before creating a new notification.

## Notification Escalation

Extends the existing notification concept for maintenance reminders.

**Fields**: notification id, recipient user id, title, message, type, reference id, reference type, read status, created timestamp, resolved or superseded status where applicable.

**Relationships**: Created from run-check findings for interview feedback, offers, simulated background checks, onboarding, and HR summary actions.

**Validation**: Recipient must be active and authorized for the referenced work item. A matching unresolved notification must prevent duplicate creation.

**State Transitions**: Open/unread -> Read; Open -> Resolved by underlying workflow; Open -> Superseded when the work item is no longer actionable.

## Archive Candidate

Represents a requisition or application eligible for archive review.

**Fields**: entity type, entity id, current status, terminal status date, related active work counts, eligibility status, eligibility reason, recommended by run-check batch, reviewed by, reviewed at.

**Relationships**: Closed requisition archive candidates relate to job requisitions and their applications. Rejected application archive candidates relate to candidate, job requisition, assessments, interviews, feedback, offers, and onboarding records.

**Validation**: Requisitions must be closed and all related applications terminal. Rejected applications must have no pending assessment, interview, feedback, offer, background-check, onboarding, or appeal work. Eligibility must be recalculated immediately before archive approval.

**State Transitions**: Recommended -> Approved; Recommended -> Blocked; Approved -> Archived; Archived -> Restored only if a future restore scope is approved.

## Archive Action

Represents an approved or blocked archive decision.

**Fields**: archive action id, entity type, entity id, action status, reason, eligibility snapshot, actor user id, action timestamp, previous active status, new archive status, affected record summary.

**Relationships**: May reference a run-check batch and archive candidate. Writes audit evidence for the affected requisition or application.

**Validation**: Actor must be active HR Admin. Reason is required. Eligibility must be valid at approval time. Archive action must not hard-delete records.

**State Transitions**: Pending review -> Approved archived; Pending review -> Blocked; Archived remains hidden from active queues and visible in archive views.

## Compliance Audit Event

Represents audit evidence for sensitive reporting and maintenance operations.

**Fields**: audit id, actor user id, actor role, entity type, entity id, action, old values, new values, reason, created timestamp.

**Relationships**: References reports, run-check batches, findings, archive actions, candidate/application/requisition records, and access-denial events where applicable.

**Validation**: Sensitive actions must include actor, action, entity type, and timestamp. Archive approvals and denials must include reason and eligibility result.
